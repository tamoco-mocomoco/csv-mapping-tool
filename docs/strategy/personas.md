# ターゲットペルソナ定義: CSV-Mapping Tool

## 概要

CSV-Mapping Toolの主要ターゲットとなる3つのペルソナを定義する。いずれも「CSVファイルのカラム構成を別のフォーマットに変換する」という共通の課題を持つが、その背景と頻度が異なる。

---

## ペルソナ 1: EC運用担当者

### プロフィール

| 項目 | 内容 |
|---|---|
| **名前** | 田中 美咲（仮名） |
| **年齢** | 32歳 |
| **役職** | ECサイト運用担当 / EC事業部 |
| **会社規模** | 中小企業（従業員50〜200名） |
| **ITスキル** | Excel中級。関数は基本的なものは使える。プログラミング経験なし |

### 業務の背景

複数のECモール（楽天市場、Amazon、Yahoo!ショッピング、自社EC）に商品を出品しており、各モールごとにCSVのカラム構成が異なる。商品マスターCSVから各モール用のCSVを作成する作業が定期的に発生する。

### 抱えている課題

- **カラム変換が煩雑**: 商品マスターのカラム名と各モールの必須カラム名が異なり、毎回Excelで手作業で並び替え・リネームしている
- **変換ミスが怖い**: 価格カラムと商品コードを取り違えるとモール側で不具合が発生し、売上に直結する
- **引き継ぎが困難**: 自分が休むと他のスタッフが変換手順を理解できず、作業が止まる
- **SJIS問題**: 楽天のCSVはSJISが必要だが、自社システムはUTF-8で出力するため、毎回文字コード変換が必要
- **時間がかかる**: 週に2〜3回、各30分〜1時間の変換作業が発生している

### CSV-Mapping Toolで解決できること

| 課題 | 解決方法 |
|---|---|
| カラム変換が煩雑 | ドラッグ&ドロップでカラムマッピングを視覚的に定義 |
| 変換ミスが怖い | プロファイル保存で毎回同じルールを適用。プレビューで確認可能 |
| 引き継ぎが困難 | プロファイルを共有すれば、誰でも同じ変換を再現できる |
| SJIS問題 | 出力エンコーディングでSJIS/UTF-8をワンクリック切替 |
| 時間がかかる | プロファイル読み込み → CSV読み込み → 変換実行の3ステップで完了 |

### このペルソナに響くメッセージ

> 「モールごとのCSV変換、もう手作業で悩まない。プロファイルを保存すれば、次からはファイルを読み込むだけ。」

## ペルソナ 2: 社内SE / 情報システム担当

### プロフィール

| 項目 | 内容 |
|---|---|
| **名前** | 鈴木 健太（仮名） |
| **年齢** | 38歳 |
| **役職** | 情報システム部 主任 |
| **会社規模** | 中堅企業（従業員200〜1,000名） |
| **ITスキル** | プログラミング経験あり（Python, VBA）。社内システムの運用・保守を担当 |

### 業務の背景

社内の各部門から「このCSVをあのシステムに取り込めるフォーマットに変換してほしい」という依頼が頻繁に来る。基幹システム、人事システム、会計システムなど、それぞれ受け入れるCSVフォーマットが異なる。

### 抱えている課題

- **変換依頼が属人化**: 自分にしかできない変換作業があり、ボトルネックになっている
- **スクリプトの保守コスト**: PythonやVBAで変換スクリプトを書いているが、仕様変更のたびに修正が必要
- **現場に渡せるツールがない**: 非エンジニアの現場担当者に「自分でやってほしい」と言いたいが、適切なツールがない
- **セキュリティ要件**: 社内データを外部サービスにアップロードすることは社内規定で禁止されている
- **予算がない**: ETLツールの導入は稟議が通らない

### CSV-Mapping Toolで解決できること

| 課題 | 解決方法 |
|---|---|
| 変換依頼が属人化 | プロファイルを作成して現場担当者に渡せば、自分で変換できるようになる |
| スクリプトの保守コスト | GUIでコンバーターを設定するだけ。コード不要で仕様変更にも即対応 |
| 現場に渡せるツールがない | ブラウザで開くだけで使えるため、非エンジニアでも導入障壁がない |
| セキュリティ要件 | ブラウザ内で全処理が完結し、データが外部に送信されないため社内規定をクリア |
| 予算がない | 無料・OSSのため、稟議不要で即導入可能 |

### このペルソナに響くメッセージ

> 「CSV変換の依頼、もう自分で抱え込まない。プロファイルを渡して現場に任せよう。ブラウザだけで使えて、データは外に出ない。」

## ペルソナ 3: データ移行担当者

### プロフィール

| 項目 | 内容 |
|---|---|
| **名前** | 高橋 亮介（仮名） |
| **年齢** | 45歳 |
| **役職** | プロジェクトマネージャー / SIer |
| **会社規模** | SIer・ITコンサルティング企業（従業員500〜5,000名） |
| **ITスキル** | PM経験豊富。技術的な理解はあるが、自分でコードを書くことは少ない |

### 業務の背景

顧客企業のシステムリプレイス案件で、旧システムから新システムへのデータ移行を担当している。旧システムからエクスポートしたCSVのカラム構成を、新システムのインポート仕様に合わせて変換する必要がある。

### 抱えている課題

- **移行仕様の確認が頻繁**: 顧客との仕様調整の中で「このカラムはこう変換して」という要件が何度も変わる
- **変換ルールの可視化**: 変換仕様書をExcelで書いているが、実際の変換結果と乖離することがある
- **テストデータの準備**: 移行テストのたびに手作業でCSVを加工しており、時間がかかる
- **複数フォーマットの管理**: 顧客ごと、テーブルごとに異なるマッピングルールがあり、管理が煩雑
- **一時的な利用**: プロジェクト期間中だけ使いたいので、ツールの導入・契約は避けたい

### CSV-Mapping Toolで解決できること

| 課題 | 解決方法 |
|---|---|
| 移行仕様の確認が頻繁 | GUIでマッピングを即変更可能。仕様変更にリアルタイムで対応できる |
| 変換ルールの可視化 | マッピングUIが変換仕様そのものとなり、仕様書と実装の乖離が起きない |
| テストデータの準備 | CSVを読み込んでプレビュー確認、即エクスポートで移行テストを高速化 |
| 複数フォーマットの管理 | テーブルごとにプロファイルを作成し、名前付きで保存・管理できる |
| 一時的な利用 | インストール不要・無料のため、プロジェクト終了後も後始末不要 |

### このペルソナに響くメッセージ

> 「データ移行のマッピング定義がそのまま変換ツールになる。仕様変更も即反映、テストデータも即生成。プロジェクト終了後の解約手続きも不要。」

## ペルソナ比較サマリー

| 項目 | EC運用担当者 | 社内SE / 情シス | データ移行担当者 |
|---|---|---|---|
| **利用頻度** | 週2〜3回（定常業務） | 月数回（依頼ベース） | プロジェクト期間中（集中的） |
| **最重視する価値** | 作業時間の短縮 | 現場への権限委譲 | 仕様変更への即応性 |
| **刺さる機能** | プロファイル保存、SJIS対応 | ローカル処理、プロファイル共有 | GUIマッピング、プレビュー |
| **導入の決め手** | 無料・簡単・繰り返し使える | データが外に出ない・非エンジニアでも使える | インストール不要・一時利用に最適 |
| **離脱リスク** | 大量データ処理の限界 | 自動化（スケジュール実行）の要望 | DB直接接続の要望 |

---

## 開発への示唆

各ペルソナの「離脱リスク」から、将来的な機能拡張の優先度が見える。

1. **大規模ファイル対応の強化** -- EC運用担当者の離脱を防ぐ（Web Worker活用等）
2. **プロファイルのエクスポート/インポート** -- 社内SEが現場にプロファイルを配布するユースケースの強化
3. **バッチ処理・CLI版の提供** -- 社内SEの自動化ニーズへの対応（将来検討）
