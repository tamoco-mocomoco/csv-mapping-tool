## はじめに ― データ移行、だいたいCSVが絡んでくる問題

システムリプレイスのプロジェクトが決まったとき、真っ先に頭をよぎるのが **「既存データの移行、どうしよう」** という問題ではないでしょうか。

旧システムからエクスポートしたCSVを、新システムのインポート仕様に合わせて変換する。文字で書くと1行ですが、実際にやってみると地味に大変です。

- カラム名が日本語 ↔ 英語で違う
- 日付のフォーマットが `YYYY/MM/DD` だったり `YYYY-MM-DDThh:mm:ss+09:00` だったり
- ステータスのコード値が `1/0` か `有効/無効` かで異なる
- 顧客番号の桁数ルールが違う

「Excelで関数組めばいけるでしょ」と思ったあなた。数百件ならそれでもいいかもしれません。でも数万件のデータを、複数テーブル分、何度もやり直す可能性がある状況で、手作業のExcel加工を続けますか？

## 手作業CSV変換の落とし穴

実際の移行プロジェクトで起きがちなトラブルを挙げてみます。

**1. ヒューマンエラーが検知しづらい**
Excel関数のコピー範囲がズレていた、VLOOKUP先のマスタが古かった――こうしたミスは目視チェックでは見つけにくく、本番投入後に発覚するケースもあります。

**2. 手順の属人化**
「変換手順はAさんのローカルのExcelに書いてある」状態になりがちです。Aさんが休んだ日にやり直しが発生すると、誰も再現できません。

**3. セキュリティリスク**
顧客データを含むCSVを「ちょっと便利だから」とオンラインの変換サービスにアップロードしてしまう。情シス的には最も避けたいパターンです。

## 具体例：旧システム → 新システムのCSV変換

ここからは、ありがちな変換パターンを4つ紹介します。

### 例1. カラム名の変換

旧システムのCSVヘッダが英語、新システムは日本語というケース。

```
旧: customer_name, email, status
新: 顧客名, メールアドレス, ステータス
```

CSV-Mapping Toolでは、出力側のカラム名を自由に設定できるので、ドラッグ＆ドロップ感覚でマッピングを定義するだけです。カラムの並び順の変更や、不要カラムの除外も同時に行えます。

### 例2. 日付フォーマットの変換

旧システムが `2025/03/28` 形式、新システムが ISO 8601 の `2025-03-28T00:00:00+09:00` 形式を要求するケース。

**日付フォーマットコンバーター** を使えば、入力・出力のフォーマットを指定するだけで一括変換できます。月オフセット（会計年度のズレ対応）にも対応しているので、「3月データを翌月として扱いたい」といった要件も設定ひとつで解決します。

### 例3. コード値の条件変換

旧システムではステータスが数値 (`1`, `0`)、新システムでは文字列 (`有効`, `無効`) というケース。

**条件代入コンバーター** で以下のようにルールを定義します。

```
値が "1" → "有効"
値が "0" → "無効"
上記以外  → "不明"
```

Excelの `IF` 関数をネストするよりも圧倒的に見通しがよく、条件の追加・変更も容易です。

### 例4. 顧客番号のゼロ埋め（パディング）

旧システムでは `12345` だった顧客番号を、新システムでは8桁固定の `00012345` にしたいケース。

**パディングコンバーター** で「8桁・左ゼロ埋め」を指定すれば完了です。データの桁数がバラバラでも、すべて統一されます。

## プロファイル保存でチーム運用

CSV-Mapping Toolの地味に嬉しい機能が **プロファイルの保存・共有** です。

一度作ったマッピング定義はJSON形式でエクスポートできます。これをGitリポジトリやファイルサーバーに置いておけば、チームメンバーの誰でも同じ変換を再現可能です。

```
移行手順書に書くこと：
1. CSV-Mapping Tool を開く
2. プロファイル「顧客マスタ_v2.json」を読み込む
3. 旧システムからエクスポートしたCSVをドロップする
4. 変換結果をダウンロードする
```

属人化とは無縁の、再現可能な手順が出来上がります。
